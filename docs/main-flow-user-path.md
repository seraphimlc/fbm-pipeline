# Main Flow User Path

状态：历史用户路径/候选背景；当前页面入口和用户路径以 `docs/domain-index/frontend-pages.md`、`docs/domain-index/product-flow.md` 和代码为准。

日期：2026-06-05
Owner：清秋（agentKey: `qingqiu`）

本文定义当前 P0 的页面体验、信息架构、用户操作路径和状态表达。范围只覆盖“商品拉取到导出任务结果文件”的主链路；A+、多店铺 ASIN、价格写入策略和新增自动化不在当前 P0 主线。

## 用户路径

### 1. 商品工作台 `/products`

用户目标：找到需要处理的商品，理解下一步动作。

页面应表达：

- 商品来自 Product 草稿，不直接展示 GIGA raw/source 半成品。
- 拉取商品是任务动作，提交后去任务中心查看进度。
- 行状态按“当前需要谁做什么”表达，例如待确认图片、待选参考竞品、抓取竞品中、待生成文案、生成中、待导出、挂起、失败。
- 待导出商品的主动作是进入导出中心或查看详情；删除、重新开始流程等危险动作不能盖过主路径。

交互边界：

- 不在商品工作台自动创建 Amazon 导出文件。
- 不把库存 0、A+ 缺失、历史已导出文件表达成商品主流程终止。
- 运行中、中断、挂起、失败必须有不同表达：运行中是系统正在执行；中断是后台动作已丢失，需要用户重跑或恢复；挂起是用户主动暂停；失败是系统已落失败事实。

### 2. 商品详情 `/products/{id}`

用户目标：处理单个商品，完成选图、竞品、类目、Listing/图片分析，进入待导出。

页面应表达：

- 首屏先渲染商品基础事实和当前步骤。
- Amazon 候选竞品、A+、其它慢请求不能阻塞首屏；失败时只影响对应 Tab，并提供重试入口。
- 类目来源归属选竞品/抓竞品详情链路，导出中心不承担常规类目确定。
- Listing 和图片分析完成后，商品可以进入待导出。
- A+ 是后续能力：商品进入待导出后可单个生成或批量生成，但 A+ 缺失不阻断当前导出主线。

交互边界：

- 未完成的 pipeline 节点不能表达成“可运营完成”。
- 竞品抓取中不能允许重复抓取或重复创建同一动作。
- 下游重新生成或切换竞品后，页面必须让用户知道哪些结果需要重新确认；但不要新增“导出过期”状态。

### 3. 导出中心 `/export-center`

用户目标：人工创建导出任务，查看历史任务和下载产物。

页面应表达：

- 导出中心是任务工作台，不是商品资格审查页。
- “待导出”表示商品已进入导出入口。
- “已导出”表示已有历史导出任务或历史文件可追溯，不表示永久禁止再次创建新导出任务。
- 模板覆盖说明模板资源是否存在；风险检查/任务结果说明这次执行中哪些行成功、跳过、失败。
- 历史产物只用于下载和追溯，不作为状态主轴。

交互边界：

- 允许用户人工创建新的导出任务；同一活跃任务防重复，同一任务/step 保持幂等。
- 已有真实 ASIN 仍按当前阶段规则禁止生成 Amazon 首次导入表，但原因必须进入任务结果或报告，而不是只靠页面前置文案。
- 库存 0、模板异常、字段异常等都应进入导出任务结果和报告。
- `partial_failed` 不能被表达成“已完成可运营”；它表示有成功产物，也有跳过或失败行，页面必须能下载产物并查看原因。

### 4. 任务中心 `/offline-tasks`

用户目标：看到任务真实状态、重跑失败动作、下载结果、追溯原因。

页面应表达：

- 任务状态是事实源：pending、running、paused、interrupted、done、failed、partial_failed。
- 导出任务的 `result_json.rows` 是页面状态的主要事实源，行状态使用稳定枚举 `exported / skipped / failed`，页面再映射中文。
- `partial_failed` 有 zip 时允许下载，同时显示成功、跳过、失败计数和逐商品原因。
- 全失败或全跳过时，即使没有产物，也应尽量保留逐商品原因。

交互边界：

- 已完成导出任务只保留下载和查看详情，不暗示旧任务会被强制“重生”。
- 失败、中断、部分失败可以提供重跑入口，但重跑应创建或恢复明确任务路径，不覆盖旧文件和旧事实。
- 页面不能只靠 toast 表达失败原因。

### 5. 库存同步 `/inventory-sync`

用户目标：理解当前库存事实和同步状态。

页面应表达：

- GIGA 库存是当前库存事实；库存 0 不阻断商品拉取到待导出主流程。
- 如果导出执行时库存仍为 0，该商品仍生成首次导入表，Quantity 写入 `0`，并在 result/report 中保留库存事实。
- 空状态需要区分从未同步、当前筛选无结果、同步任务失败或同步中。
- 提供去任务中心查看最近库存同步任务的路径。

交互边界：

- 库存页不判断商品是否可永久铺货。
- 库存事实不能被表达成运营决策；补货、继续导出或暂缓属于用户决策。

## 状态语言

页面必须区分四类信息：

- 数据事实：库存数量、是否有真实 ASIN、是否有历史导出文件、当前任务状态、逐商品导出结果。
- 系统建议：建议重跑、建议查看任务中心、建议补模板、建议复核字段。
- 用户决策：选择竞品、确认图片、创建导出任务、下载文件、是否继续处理库存 0 商品。
- 运营风险：真实 ASIN 拦截首次导入表、库存 0 但继续导出、模板缺失、字段异常、导出部分失败。

禁止状态混用：

- 真实 ASIN 不是导出状态。
- 历史已导出不是禁止再次新建导出任务。
- 库存 0 不是主流程永久终止。
- A+ 缺失不是当前 P0 导出阻断。
- `done` 不等于可运营完成；Step 10 和导出任务只生成文件与风险提示，仍需人工确认。
- `partial_failed` 不等于完成；必须暴露失败/跳过原因。

## 当前 UI 债

- `ProductDetail` 首屏不应被 `/api/amazon-stylesnap/products/{id}/competitor-candidates` 等慢请求卡住。
- `CatalogList` 仍存在“已导出商品只用于查看，不能再次生成 Amazon 导入表格”的旧文案和硬阻断，需要按任务工作台口径修正。
- `OfflineTaskCenter` 对完成导出任务不应展示会误导用户的禁用“重跑”按钮；完成任务应突出下载和详情。
- `AplusManagement` 不能用 `amazon_asin` 判断导出状态；A+ 当前作为后续体验债，不纳入 TT-090 P0 主线。
- `InventorySyncList` 空状态需要说明从未同步、筛选为空、同步失败或同步中。
- 商品工作台待导出行需要弱化危险操作，把“去导出中心/查看详情”作为主路径。

## 验收标准

- 用户能从 `/products` 发起或追踪拉品任务，并在 `/offline-tasks` 看到明确结果。
- 用户能进入 `/products/{id}`，首屏不因非首屏请求 pending 而卡死。
- 用户能完成或复核选图、搜索候选竞品、选择竞品、抓竞品详情、Listing/图片分析。
- 商品进入待导出后，用户能在 `/export-center` 创建导出任务。
- 导出任务结果能在 `/offline-tasks` 或 `/export-center` 看到成功、跳过、失败、部分失败和下载入口。
- 历史文件保留下载和追溯；新任务生成新文件，不覆盖旧事实。
- 页面不把未完成 pipeline、部分失败、库存 0、A+ 缺失或历史产物表达成“可运营完成”。
