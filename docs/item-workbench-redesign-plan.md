# Item 商品工作台重构规划

日期：2026-06-03

## 目标

把原来的“大健商品池 / 同款抽卡 / 商品生成任务 / 商品列表”收敛成 Item 维度的商品工作台。

第一版目标不是全自动，而是跑通高质量商品数据生产线：所有关键节点都人工确认，系统只执行已经确认的自动动作。

## 主对象边界

- 主流程对象是大健 Item，一行一个 Item。
- SKU 不作为主流程对象，只作为 Item 下的变体明细。
- SKU 维度保留：SKU 编码、变体属性、SKU 图片、价格、库存。
- Item 维度保留：参考竞品、类目、标题、五点、描述、Listing 图片、A+、导出资格。
- 一个 Item 只选择一个 Amazon 参考竞品。竞品用于标题、五点、描述、类目和买家关注点参考。

## 数据层边界

- GIGA 来源层：`giga_sync_batches`、`giga_raw_sku_details`、`giga_items`、`giga_skus`、`giga_product_images`、`giga_prices`、`giga_inventory`，按 `batch_id + site` 保存原始/事实数据。
- 商品数据源层：`product_data_sources` 维护来源店铺/账号，包含平台、站点、AK/SK 和履约方式。新同步数据同时带 `data_source_id`，用于区分多个 US/JP 店铺；运费和库存口径由履约方式在系统内部推导。
- Amazon 参考层：`amazon_stylesnap_candidates` 保存 Top 5 候选，`amazon_listing_captures` 保存选中竞品的 Listing 抓取结果。
- 正式商品任务层：`products`、`product_data`、`product_images`、`product_aplus` 保存合并处理后用于生成 Listing/A+/导出的商品资料。
- 商品工作台只展示 Product/SKU 维度流程；GIGA 等 raw/source 表只作为采集输入和追溯证据，不作为用户主操作对象。

## 第一版状态机

业务状态按“当前需要谁做什么”命名：

1. 待补基础数据
2. 待确认商品图片
3. 找竞品中
4. 待选参考竞品
5. 抓取竞品中
6. 待生成文案
7. 生成文案中
8. 待复核文案
9. 生成 A+ 中
10. 待复核 A+
11. 待导出
12. 已导出
13. 已上架
14. 挂起

第一阶段先用现有 `products.status` 和 `current_step` 映射前台业务状态，不立即做大规模数据库迁移。
其中“挂起”复用底层 `products.status = paused`：它不是失败状态，也不是人工复核节点，而是用户明确要求停止后续未完成自动流程的状态。

## 人工节点

第一版全部关键节点人工确认：

- 确认商品图片：一次性选择用于 Amazon 搜图的主图，并确认 Listing 主图/副图顺序。
- 选择参考竞品：从 Top 5 里选一个 Item 级参考竞品。
- 竞品抓取中：系统正在通过 Chrome 抓取 Amazon Listing，页面只说明当前动作，不允许重复抓取、创建任务或重跑。
- 确认生成文案：确认基础数据和竞品抓取结果后再启动文案生成。
- 复核文案：系统生成后人工确认，不做人工编辑。
- 复核 A+：系统生成后人工确认，不满意就 redo。
- 导出：导出中心按类目批量导出。
- 挂起/继续：任意已创建的正式商品任务都可以挂起；挂起后不再执行后续自动流程，点击继续后从保存的 `current_step` 恢复。

## Redo 规则

第一版不做人工作文案编辑，只支持人工触发 redo。

- 重新找竞品：回到待选参考竞品，下游文案、图片、A+、导出状态失效。
- 重新抓取竞品 Listing：回到待生成文案，下游文案、A+、导出状态失效。
- 重新生成文案：回到待复核文案，A+ 和导出状态失效。
- 重新选择 Listing 图片：回到待选择 Listing 图片，A+ 和导出状态失效。
- 重新生成 A+：回到待复核 A+，导出状态失效。
- SKU 价格、库存、UPC、属性变化：回到待导出，导出状态失效。

后续应拆出独立 `workflow_status` 和 `export_status`：

- `workflow_status` 表示当前商品生产节点。
- `export_status` 表示导出文件是否未导出、已导出或已过期。

## 页面收敛

一级菜单第一版收敛为：

- 商品工作台
- 任务中心
- 导出中心
- 库存同步
- ASIN 同步
- A+ 上传
- UPC 池子
- 数据源维护
- 系统配置

原入口处理：

- 大健商品池：收进商品工作台，作为来源数据/待补数据视图。
- 同款抽卡：收进商品工作台，作为 Item 行内动作。
- 商品生成任务：收进商品工作台，作为 Item 状态和批量操作。
- 商品列表：改为导出中心，后续按待导出 Item 聚合。

## 导出中心边界

导出不放在单品详情，也不放在优化 pipeline 里。

导出中心后续按类目分组：

- 查询所有待导出 Item。
- 按 Amazon 类目分组。
- 有模板的类目可导出。
- 缺模板的类目挂起，等待上传模板。
- 导出全部时只导出模板已就绪的类目，缺模板类目不阻塞。

模板适配后续分三种：

- 通用映射 JSON。
- 类目专用 Python 脚本。
- 模型辅助识别和校验映射，但不作为第一版强依赖。

## 当前第一阶段落地

- 菜单已收敛，隐藏大健商品池、同款抽卡、手动创建任务的一级入口。
- 已新增商品数据源维护入口，用于配置大健云仓等来源店铺的 AK/SK、站点和履约方式。
- 商品工作台复用现有 ProductList，只显示已标准化的 Product。
- 商品工作台查询不要求用户选择批次；批次只作为内部来源字段保留，页面按状态和 SKU Code 过滤。
- 商品工作台提供“拉取大健商品”按钮，先选择商品数据源再增量拉取：库里已有 SKU 跳过，只把缺失 SKU 拉回 GIGA 来源层。
- 商品工作台拉取大健商品已改为提交到任务中心：页面必须选择商品数据源，支持多选；后台创建一个父任务并按数据源分别执行。
- GIGA 商品图片先保存 URL 作为 Product 图片候选；确认商品图片时只按需下载已选主图，后续图片分析/A+等节点需要本地素材时再按需下载。
- GIGA Item 分组完成后自动 upsert Product 草稿，状态为“待确认商品图片”；商品工作台不再显示 GIGA/raw 半成品行。
- 商品工作台已显示待选参考竞品、待抓 Listing、待确认生成、生成中、待复核/待导出等状态。
- 商品工作台已补充竞品抓取中状态，抓取期间只提示等待，禁止重复操作。
- 商品工作台已补充挂起状态，已挂起商品不会继续跑自动流程，只能由用户点击继续恢复。
- 待选参考竞品可在商品工作台打开 Top 5 候选弹窗，选择一个 Item 级 Amazon 参考竞品。
- 已选参考竞品但未抓 Listing 的 Item，可在商品工作台行内触发抓取 Listing。
- 商品草稿来自 GIGA 分组后的 Product upsert；参考竞品只负责进入 Product 的竞品选择结果，不再通过竞品候选反向生成商品。
- 商品优化 pipeline 停在 Step9 A+ 出图，不再自动生成 Amazon 导入表格。
- 商品详情确认按钮语义改为加入待导出。

## 自动任务建议

当前后台执行不是定时查表，而是 API 触发后通过任务中心创建 `offline_tasks` / `offline_task_steps`，再用 `asyncio.create_task()` 启动内存执行；服务启动时把运行中离线任务标记为中断，等待人工重跑，不恢复 `paused` 商品。

第一版任务中心已经承载：

- 商品中心多数据源拉品。
- 确认商品图片后的主图按需下载；不再拉品后全量下载所有商品图片。
- 失败/中断任务的重跑入口。

后续 Excel 导出、Amazon Listing 抓取、A+ 生成/重生成也应逐步接入任务中心。

后续如果要做更自动化的流程，建议新增一个轻量后台 worker：

- 定时查询“系统可以自动推进”的状态，例如已确认商品图片、已选竞品待抓 Listing、待生成文案。
- 查询条件必须排除 `products.status = paused`，并且排除竞品抓取中、A+重新生成中等中间状态。
- worker 每次只领取少量任务，先用状态条件做原子更新，再启动具体动作，避免重复领取。
- 所有动作保持幂等：再次执行时先检查已有候选、已选竞品、已抓 Listing、已生成内容。
- 人工节点仍停在工作台，由用户确认后再让 worker 或接口继续。

第一版可以继续保留“用户点击触发 + 启动恢复运行中任务”的方式；等流程跑稳定后，再加这个定时 worker。这样自动化不会绕过人工确认，也不会把挂起商品重新唤醒。

## 商品数据源

商品数据源不是商品流程结点，而是拉品和后续成本/库存口径的来源配置。

第一版字段：

- 平台：当前先支持 `giga`。
- 站点：如 `US`、`JP`，这里同时代表国家/站点概念。
- 履约方式：`self_ship` 自发货、`dropship` 代发货。
- 运费和库存口径：不在页面维护，由履约方式自动推导；自发货按库存分布和小额打包成本处理，代发货按 GIGA 可售数量和真实运费处理。
- 凭据：每个数据源维护独立 AK/SK，前端只展示脱敏 SK。

同步边界：

- 商品工作台拉品必须选择数据源。
- 新 GIGA 同步记录、Item、SKU、图片、价格、库存都会带 `data_source_id`。
- 旧数据仍保留 `site` 兼容展示；后续稳定后可做一次专门迁移，把唯一约束升级为 `data_source_id + site + sku/item`。
