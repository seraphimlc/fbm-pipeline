# Full Audit Playbook

适用角色：镜花为主；若命在发起全量审计任务时也应按需读取。

读取条件：

- 中间已有提交，镜花没有参与每轮变更。
- 用户要求“从头到尾 review”或“全量审计”。
- 系统出现连续质量问题，diff 已不足以解释当前风险。
- 模块长期没人审，或需要建立一个领域的代码质量基线。

## 核心原则

全量审计不以“本轮 diff”为边界，而以审计范围为边界。它是只读审计任务：先规划，再拆项，再逐项执行，再汇总报告。不能边看边随口下结论。

## 执行流程

1. 定义审计范围：明确审哪个业务域、模块、页面/API、数据链路、任务流或发布链路；同时写清不审范围。
2. 建立系统地图：先读 `docs/project-index.md` 和相关 `docs/domain-index/*.md`，再用代码事实校验入口、路由/API、service/domain、repository/DAO、worker/action、数据表、外部依赖、前端页面、文档、验证入口和测试位置。
3. 读取历史线索：查看相关 PRD/spec、inbox、review、QA、近期 commit、git log、git blame 或变更记录；不能假设自己知道中间发生了什么。
4. 制定审计计划：把审计拆成有顺序、可验证的审计项，类似任务规划一样管理。
5. 建立质量基线：列出该范围必须满足的结构、状态、数据、查询、事务、错误、文档和测试不变量。
6. 逐项审计：按审计计划一项一项执行，从用户/API 入口走到数据落库、任务执行、外部调用、结果返回和页面展示；记录真实依赖方向和副作用。
7. 抽样验证：选择高风险路径做静态检查、只读 API/DB 样本、最小复现或命令验证；全量审计可以分阶段输出。
8. 输出审计报告：按 P0/P1/P2 分组，写系统性问题、局部问题、证据、修复顺序、建议拆分任务和未覆盖风险。

## 执行可见性

- 大型审计必须先给若命/用户发 `AUDIT_PLAN`，或写入审计文件并在 inbox 留链接，等范围清楚后再执行。
- 执行过程中按审计项更新状态：`pending / in_progress / done / blocked`。
- 每完成一个审计项，至少记录：审了哪些文件/路径、使用了什么证据、发现了什么、未覆盖什么。
- 如果审计超过一轮会话或需要分阶段完成，写阶段性 `STATUS`，不要让审计过程变成黑盒。
- 审计过程中发现 P0 时可以先发短消息阻断风险，再继续完成剩余审计项。

## 只读和子 agent 规则

- 全量审计默认是只读操作。镜花不得修改业务代码、真实数据、生成产物、配置、任务状态或外部平台状态，除非用户明确授权。
- 允许运行只读命令、静态分析、测试命令、只读 API/DB 查询和最小复现。
- 任何写操作、迁移、任务触发、导出、上传、状态推进都必须先确认授权。
- 镜花可以启动子 agent 协助做只读专项审计，例如代码结构、数据模型、查询性能、测试质量、文档一致性、安全边界。
- 子 agent 必须有明确范围、只读边界、输出格式和停止条件；不能让子 agent 自行扩大审计范围。
- 镜花负责汇总子 agent 结果，去重、判级、校验证据和形成最终结论；不能把子 agent 原始结论直接当最终结论。

## 审计要求

- 不因为没有本轮 diff 就放弃审查；用代码、历史提交和运行事实重建上下文。
- 不因为范围大就泛泛而谈；必须按模块/功能/调用链拆分。
- 不把所有问题一次性丢给执行者；要给修复优先级和分批边界。
- 不给“整体还行”这种空泛结论；必须说明审了什么、没审什么、发现什么、下一步谁做什么。
- 如果范围过大无法一次完成，写 `CODE_REVIEW / BLOCKED` 或阶段性审计计划，先收窄范围再继续。

## 报告模板

大任务、全量审计、跨模块审计、需要多轮执行或使用子 agent 的审计，必须生成报告文件，建议放到 `docs/collaboration/reviews/`，并在 inbox 回复摘要和报告链接。

```markdown
# CODE_AUDIT_REPORT - <范围>

- Reviewer: 镜花（agentKey: `jinghua`）
- Trigger:
- Scope:
- Out of scope:
- Read-only statement:
- Sub-agents used:
- Summary:
- Audit plan and progress:
- System map:
- Index review:
- Findings:
- Passed checks:
- Documentation impact:
- Test and verification evidence:
- Uncovered areas:
- Recommended repair order:
- Next actions:
```

报告不能只列问题，还要列审计范围、审计计划、执行进度、证据来源、未覆盖范围和下一步建议。审计被阻塞时，也要回复 `CODE_REVIEW / BLOCKED`，说明阻塞原因、已完成审计项、还缺什么输入。
