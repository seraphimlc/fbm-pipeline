# Context Indexing Playbook

适用角色：所有角色。若命派工、听云执行、镜花 review、观止 QA、清秋走查、霜弦复核时都应按需使用。

## 读取条件

- 新会话冷启动，需要快速理解项目结构。
- 用户提出的问题需要定位代码、页面、API、数据表或文档。
- 多 agent 反复探索同一模块，token 消耗偏高。
- 新增/修改页面、API、任务类型、状态机、数据表、导出链路或外部集成。
- review/QA 发现索引过期、缺失或误导。

## 核心原则

索引不是事实源，而是导航层。代码、数据库只读事实、API、页面行为、命令输出和用户明确口径仍然是事实源。

索引只回答“这个问题应该去哪里看”，不回答“完整怎么设计、为什么这么设计、历史上怎么讨论”。

正确流程：

```text
用户问题
-> docs/project-index.md 定位领域
-> docs/domain-index/*.md 找关键入口
-> git status --short 确认当前工作区事实
-> rg 在候选范围内核实
-> 只读关键文件片段
-> 执行/验证
-> 必要时更新索引
```

## 索引层级

- `AGENTS.md`：项目硬规则和安全边界。
- `docs/project-index.md`：项目总地图，负责“问题类型 -> 领域地图 -> 关键入口”。
- `docs/domain-index/*.md`：领域地图，负责页面/API/service/model/table/docs/验证入口。
- `docs/collaboration.md`：多 agent 协作入口。
- `docs/collaboration/playbooks/*.md`：复杂执行手册。
- `docs/superpowers/specs/*.md` 或项目约定 design/spec：详细设计和 PRD。

## 项目地图初始化

新项目刚完成 `multi-agent-collaboration` 初始化后，第一阶段建议先做“项目地图初始化”，再进入具体功能开发。

目标不是写完整项目说明书，而是建立 `project-index -> domain-index -> scoped rg -> 关键片段` 的稳定工作路径，减少后续任务漏看页面、API、service、表、任务、验证入口或相邻领域的风险。

推荐分工：

- 若命：负责分域、边界、命名、成功标准和哪些内容不进地图。
- 听云：负责代码事实扫描，补页面/API/service/model/table/task/config/test/验证入口，不急着改代码。
- 镜花：负责审地图是否漏入口、误导、过细、过大或把设计过程写进地图。
- 观止：负责补用户路径、验收入口、关键页面/API 样本和 QA 可用验证路径。

写入规则：

- 地图可以承接设计沉淀，但只承接“索引化后的结果”。
- 完整设计、取舍、状态机细节、字段定义、迁移方案写到 PRD/spec，不写进地图。
- 跨 agent 行动仍写 inbox 顶层消息，不藏在地图里。
- 地图初始化完成后，后续任务默认从 `project-index -> domain-index -> scoped rg -> 关键片段` 开始。

## project-index 边界

`docs/project-index.md` 只做总路由：

- 问题类型。
- 应先读哪个 domain index。
- 最关键的 1-3 个入口。

不要把领域细节、历史讨论、任务过程或长说明放进总地图。

## domain-index 固定模板

领域地图只能使用以下栏目，避免失控：

```markdown
# Domain Index: <Name>

## 范围
## 当前口径
## 关键入口
## 关键流程
## 相关文档
## 验证入口
## 常见定位
## 维护规则
```

领域地图只保留三类信息：

- 去哪看：页面、API、service、worker/action、表、配置、验证入口。
- 别走错：范围、不包含什么、当前稳定口径。
- 怎么继续查：PRD/spec/review/QA/handoff 链接和简短定位指引。

## 内容禁止

领域地图不要写：

- 普通 bug 修复记录。
- 长设计解释、完整字段表、长 SQL、代码片段。
- 历史聊天、协作过程、任务列表、争议过程。
- 长测试输出、完整导出样例、完整数据样本。
- 临时文件、临时脚本、一次性调试命令。
- 不稳定猜测或“某某说过”。

## 长度约束

- `范围` 最多 5 条。
- `当前口径` 最多 8 条。
- `关键流程` 最多 5 条，每条一行。
- `相关文档` 最多 10 个链接。
- `验证入口` 最多 8 条。
- `常见定位` 最多 8 条。
- 单个领域地图建议不超过 150 行。

超过限制时，不要继续堆内容；应把详细信息放到 PRD/spec/review/QA/handoff，并在领域地图里只放链接。

## 各角色使用方式

- 若命：派工前先查 `docs/project-index.md`，在 REQUEST 中写明相关 domain index 和关键文件候选。
- 听云：开工前读 REQUEST 指定的 domain index，用 `git status --short` 和 scoped `rg` 核实当前实现；变更影响索引时负责更新。
- 镜花：review 前用索引建立模块边界；全量审计时把“索引是否准确”列为审计项；索引缺失和索引污染都要指出。
- 观止：QA 前用索引确认页面、API、数据表、副作用和验证入口。
- 清秋：走查前用索引确认页面入口和用户路径。
- 霜弦：复核运营/数据规则前用索引找到事实源。

## 维护触发

必须更新索引：

- 新增、删除、重命名页面、API、service、worker/action、任务类型、数据表。
- 状态机、操作矩阵、分页/统计口径、导出链路、外部集成或验证入口发生变化。
- 详细设计或 PRD 替代了旧实现口径。
- review/QA 发现索引过期、缺失或误导。

可以不更新索引：

- 局部 bug fix，且入口、状态、API、数据表、验证方式都没变。
- 函数内部重构。
- 纯样式或文案微调。
- 测试补充，且不改变主要验证命令和路径。

## 写法边界

- 索引只写定位信息和当前口径摘要，不复制代码。
- 索引不放长日志、完整导出样例、完整聊天历史、敏感信息。
- 索引不能替代 PRD/spec；复杂设计仍写到 design/spec。
- 索引不能替代 inbox；需要 agent 行动仍写顶层 message。
