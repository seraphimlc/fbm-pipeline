# QA Playbook

适用角色：观止为主；若命在下发 QA gate、听云准备 DONE_CLAIMED 时可按需读取。

读取条件：

- 正式 QA gate、回归、发布前 smoke、阻塞复现或外部平台验证。
- QA 任务需要白盒验证、测试矩阵、证据索引或场景化验收。
- 需要判断 `PASS / NEEDS_FIX / BLOCKED`。

## 白盒验收要求

- 必须读实现相关文件，至少定位到涉及的 API、服务、任务 worker、状态派生、前端页面和测试规则。
- 必须核对 PRD/REQUEST 里的字段、状态、按钮、流程、错误、边界和禁止范围是否在代码里真实落地。
- 必须区分施工者声明、代码事实、运行事实和用户事实；听云的 `DONE_CLAIMED` 只是线索，不是证据。
- 必须优先用只读事实验证：GET API、只读 SQL、页面观察、构建/测试命令、生成产物抽查、日志片段、截图路径。
- 涉及数据库写入、任务触发、导出、真实业务状态、生成产物或外部平台操作时，先确认 QA 任务是否授权；未授权只做只读验证。
- 发现实现和 PRD 口径不一致时，不替听云解释合理性；写清 PRD 期望、代码实际、用户影响和最小复现。

## 测试用例设计

每次 QA 至少先写出一组轻量测试矩阵，可写在 inbox 的 REVIEW 中，也可写到 review/QA 文件后在 inbox 留链接。

测试矩阵应包含：

- 用例 ID：稳定命名，如 `TC-TASK-001`。
- 验证目标：本用例证明哪个 PRD 条款或 bug 修复。
- 前置条件：账号、页面、数据源、任务状态、样本 ID、环境服务。
- 操作步骤：页面点击、API 请求、DB 只读查询、命令或产物打开方式。
- 预期结果：状态、字段、按钮、错误提示、产物内容、副作用边界。
- 实际结果：观察到的事实。
- 证据：命令、URL、截图路径、API 摘要、DB 查询摘要、文件路径。
- 判定：PASS / FAIL / BLOCKED。

用例来源优先级：

1. PRD 明确验收样本和历史 bug 样本。
2. 听云 `DONE_CLAIMED` 中声称已验证的样本。
3. 用户现场指出的问题路径。
4. 代码中高风险分支：状态机、分页统计、异步任务、重试/取消、权限、幂等、错误恢复。
5. 观止补充的边界样本：空数据、大数据、失败、重复、刷新、回退、直接 URL、页面返回。

## QA 文档责任

观止负责测试和验收层面的文档留痕。不是每次点击都写长文档，但每轮正式 QA、回归、发布前 smoke、阻塞复现或外部平台验证，都必须留下可复查证据。

必须写或更新文档的情况：

- 若命派发 QA gate、听云提交 `DONE_CLAIMED` 后进入正式验收。
- 发现 P0/P1，或需要别人按步骤复现问题。
- 验证涉及真实数据、外部平台、导出/发布产物、权限、配置或不可逆副作用。
- 测试矩阵、样本选择、环境状态或未覆盖风险需要后续回归。

文档类型：

- QA plan / test matrix：写目标、范围、样本、前置条件、步骤、预期、允许/禁止副作用。
- QA review：写 PASS/NEEDS_FIX/BLOCKED、证据、P0/P1/P2、最小复现、未覆盖和下一步责任人。
- 现场证据索引：截图、API 摘要、DB 只读查询、命令输出、产物路径、日志片段只写摘要和路径。
- 回归记录：写旧问题是否复现、修复后同路径结果、相邻风险是否回归。

不要做：

- 不用“我测过了”替代测试矩阵和证据。
- 不把缺样本、缺预期、缺授权的 QA 包装成 PASS。
- 不把新任务藏在 QA review 里；需要返工时由若命或按协议新建顶层 message。

## 工作场景

不同 QA 场景不能套同一份检查单，按场景调整重点。

### 功能验收

- 先读 PRD/REQUEST，确认字段、状态、操作、页面入口、API 合同和成功标准。
- 验主路径、关键边界、错误路径和刷新/重进页面后的状态保持。
- 如果功能会产生数据，必须说明产生了什么数据、是否可回滚、是否影响真实业务。

### Bug 修复复验

- 先复现旧问题或确认旧问题在当前环境的最小触发条件。
- 再验证修复后同一路径的实际结果。
- 最后做相邻回归：同类状态、同类按钮、同类筛选、同类任务是否被修坏。
- 不能只测听云说修过的那一条 happy path。

### 任务中心 / 异步流程验收

- 核对任务创建、入队、执行中、成功、失败、部分失败、中断、取消、重试、恢复、历史归档和 superseded 语义。
- 核对列表状态和详情状态是否同源，筛选 total 和分页是否来自真实 DB 查询。
- 检查相关接口是否存在嵌套查询、`EXISTS/IN` 子查询、多表关联查询、重复 count、运行时状态推导、内存过滤分页或查询后二次拼装；发现后不能只看页面能否加载，应作为性能和设计风险写 `NEEDS_FIX`。
- 核对按钮是否只出现在允许状态，点击后是否有真实 API 动作和可追踪事件。
- 只读情况下不要启动真实大批量任务；若任务需要现场触发，必须先确认样本数量、数据源、停止条件和副作用。

### 页面交互 / UX 回归验收

- 走真实用户路径，不只检查组件是否渲染。
- 检查空状态、加载中、错误提示、慢接口、重复点击、刷新、返回、URL 参数、分页和筛选组合。
- 区分 UX 问题和系统语义问题；语义不清时转给若命，不让实现者用 UI 文案遮盖。

### 数据 / 导出 / 产物验收

- 明确样本范围，优先抽小样本。
- 只读检查 DB、导出文件、报告 rows、错误报告、下载入口、文件命名、模板字段和历史记录。
- 不覆盖真实导出文件、模板、已生成素材，不批量修改真实业务状态。
- 如果必须上传或外部平台验证，先取得用户授权，并记录账号、平台、文件、错误和结果。

### 回归 / 发布前 Smoke

- 按影响面选择核心路径，不追求无边界全量测试。
- 至少覆盖：启动、主入口页面、相关 API、核心状态筛选、关键按钮、错误提示、构建/项目规则。
- 对未覆盖模块写残余风险，不把未测范围包装成通过。

## 证据格式

观止的 REVIEW 必须让若命/用户不用重新跑一遍也能判断质量。

最少包含：

- 结论：`PASS / NEEDS_FIX / BLOCKED`，以及一句话原因。
- 范围：本轮验了哪些页面/API/文件/任务类型/样本。
- 环境：服务地址、分支/工作区状态摘要、是否启动服务、是否有 DB/API 阻塞。
- 用例摘要：列测试矩阵或链接到 QA 文件。
- 证据：命令结果摘要、API 样本、只读 DB 摘要、截图/产物路径。
- 副作用：是否触发写操作、创建任务、生成文件、修改状态；没有也要写“未触发写操作”。
- 问题：按 P0/P1/P2 排序，每条写最小复现、期望、实际、影响、证据。
- 未覆盖：没有测什么，为什么没测，风险是什么。
- 下一步：谁需要做什么；如果需要听云返工，若命应另建顶层 message。

## 判定标准

### PASS

只有同时满足以下条件才能 PASS：

- PRD/REQUEST 的验收目标已覆盖。
- 关键用户路径、关键状态和关键副作用已验证。
- 没有 P0/P1 未解决问题。
- 未覆盖项不会阻塞本轮目标，且已明确残余风险。
- 证据足够支撑结论，不依赖听云声明。

### NEEDS_FIX

出现以下任一情况应 NEEDS_FIX：

- 实现和 PRD/用户口径不一致。
- 用户可见路径失败、按钮无效、状态误导、统计不准、分页不可信。
- 数据副作用不明、任务不可追踪、失败不可恢复、重试/取消语义错误。
- 只测 happy path，关键失败/边界路径缺失且风险会影响用户。
- 听云证据和现场事实冲突。

### BLOCKED

出现以下情况应 BLOCKED：

- 环境不可用、权限缺失、登录态缺失、依赖服务不可达。
- QA 任务缺少样本、预期、授权或判定标准，且无法从 PRD/代码中合理确定。
- 继续验证会触碰未授权真实数据或外部平台。
- 必要证据缺失，且观止无法只读补齐。
